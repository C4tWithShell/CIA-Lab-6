CURL

To display the contents of a full HTTP/1.1 session I used curl -v (Figures 9, 10).


Figure 9 - Content of session for aaa


Figure 10 - Content of session for bbb

From Trying 3.140.193.56:80... to * Mark ... is request to Nginx server.
First it tried to connect to host. The TCP_NODELAY option allows you to send data as fast as it arrives.

Request

GET / HTTP/1.1 - the main page of the site is requested.
GET is a method. It defines the operation to be performed with the specified resource.
/ is URI (Uniform Resource Identifier) — the path to a specific resource.
1.1 the is version of HTTP.

Host: bbb(or aaa).st12.sne21.ru - this is the Host header, which is mandatory, and must be present in every request. The fact is that the conversion of a domain name to an IP address is carried out on the client side.

The User-Agent (UA) string is the information included in the HTTP header, acting on behalf of the user. For me, it is User-Agent: curl/7.68.0

Accept: */* - the client can understand any MIME type.

Response

Headers

HTTP/1.1 200 OK is the starting line of the response.

1.1 the is version of HTTP.

200 is the status code. 200 means that the GET method was used, and the server provides a resource with the specified identifier. If the server reports that such a resource does not exist - 404. If the server reports that it cannot provide access to this resource because the client does not have the necessary privileges, then code 403 is used. The HTTP 1.1 specification defines 40 different HTTP codes.
OK is the reason phrase - text explanation of the response code.

The start line is followed by the headers, as well as the response body.

Server header - A name for the server.
Date header - The date and time that the message was sent.

Content-Type header describes the original format of the message data and any encoding applied.
To determine the end of the response body, the value of the Content-Length header is used.
Last-Modified - The last modified date for the requested object.

Connection - Control options for the current connection and list of hop-by-hop response fields.
ETag is an identifier for a specific version of a resource.
Accept-Ranges - what partial content range types this server supports via byte serving.

Body

Self-signed key and certificate pair with OpenSSl can be created by command sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 -keyout /etc/ssl/private/nginx-selfsigned.key -out /etc/ssl/certs/nginx-selfsigned.crt

req - specifies that we want to use X.509 certificate signing request management.

-x509 - modifies req to make a self-signed certificate instead of CSR.

-nodes - no passphase.

-days 365 - it is valid for one year.

-newkey rsa:2048 - generate a new certificate and a new key (RSA key that is 2048 bits long) at the same time.

-keyout - where to place the generated private key file.

-out - where to place the certificate.

Then a few questions will be asked about our server, this information will be embedded in the certificate.

I also created a strong Diffie-Hellman group for OpenSSl - sudo openssl dhparam -out /etc/ssl/certs/dhparam.pem 2048.

I added ssl_dhparam /etc/ssl/certs/dhparam.pem; to /etc/nginx/snippets/ssl-params.conf.

FreeIPA (from Free Identity, Policy, Audit) is an open project for creating a centralized enterprise-scale system for managing user identification, setting access policies and auditing networks. It includes the following components:

  • 389 Directory Server as an LDAP server;
  • MIT Kerberos 5.0 for Authentication and Single Sign-on;
  • DogTag for Certificate Management;
  • NTP;
  • BIND and DHCP for DNS management;
  • web-based management interface.

FreeIPA also uses Samba to integrate with Microsoft’s Active Directory through a trust relationship.

FreeIPA is a specialized domain controller whose goals and mechanisms are similar to Microsoft Active Directory.

Policy management is implemented based on HBAC rules (host based access control, node–level access control - describes which services are available to users on a particular registered host) and sudo rules (which commands a user can execute as root or another user).

The system also allows you to delegate various roles to users very flexibly without passing them the administrator password. For example, you can assign a special role to the user Enroll hosts, which will allow you to register hosts.

A mechanism for building a multi-level access control system has also been implemented, that is, the head of a user group can be given the authority to add new users to it.
